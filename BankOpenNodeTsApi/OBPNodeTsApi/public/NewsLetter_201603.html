i-BANK Api
Summary
Two months after becoming the first Greek bank to open the doors to developers around the world we are now moving to a more advanced approach. We now made URLs more meaningful, we used headers for auth emulation to identify logged in users. User module now exists with association to the banks. Compatibility with OPB is now a first priority issue for us. Below we describe all the changes that took place providing examples and URLs to help you use the new API.
 
Functional Design

The functional specs at the end of the project will be the same as in openbankproject.com but are going to help you monitor the current status. Currently, “put” and “delete” are fully functional for banks, atms, branches, products, accounts and are partially functional for transactions, views and metadata.

Changes - Api calls

There is now specific logic in API calls so if we get:
api/banks we get all banks but
api/banks/:bank_id gets only one bank as a result
api/banks/:bank_id/atms we get all atms of one bank
api/banks/:bank_id/accounts/public we get all public accounts of one bank
api/accounts/public we get all public accounts

Changes - Headers

In certain calls you will see the mention “Auth authentication is required”. That means you try to access a protected resource and an Auth header is needed. For SandBox the header for these calls must have:
Auth-Provider-Name: fakelogin
Auth-ID: 123456789
Additionally, for tracking the calls and read the body you must have 2 more headers you must have 2 more headers
Content-Type: application/json
Track-ID: 572afc88ee5c07d41a8888d4
Changes - Personal rest calls that need user&customer models
So using the above headers we can emulate oauth. Plus, with the node passport, this functionality can be easily integrated with a provider (try it with SandBox!). Additionally we can track the requests and get extra parameters on any REST call. Here are some calls that need user “registration”:
PUT:api/banks That can add or change a bank
DELETE:api/banks/bank_id/atms That can delete an atm
GET:api/banks/:bank_id/accounts/private brings all personal accounts of one bank
GET:api/banks/:bank_id/customer brings customer details for user logged in
For more detailed call description visit sandbox’s public/api.json
 
Examples

Examples to run with the sandbox or from NBG api portal GET /api/banks/ HTTP/1.1
Host: localhost:3005
Cache-Control: no-cache

GET /api/banks/5710bba5d42604e4072d1e72 HTTP/1.1
Host: localhost:3005
Cache-Control: no-cache

GET /api/user HTTP/1.1
Host: localhost:3005
Auth-Provider-Name: fakelogin
Auth-ID: 123456789
Track-ID: 5710bba5d42604e4072d1e72
Cache-Control: no-cache

GET /api/banks/5710bba5d42604e4072d1e72/customer HTTP/1.1
Host: localhost:3005
Content-Type: application/json
Track-ID: 572afc88ee5c07d41a8888d4
Auth-Provider-Name: fakelogin
Auth-ID: 123456789
Cache-Control: no-cache

GET /api/banks/5710bba5d42604e4072d1e72/accounts HTTP/1.1
Host: localhost:3005
Content-Type: application/json
Track-ID: 572afc88ee5c07d41a8888d4
Auth-Provider-Name: fakelogin
Auth-ID: 123456789
Cache-Control: no-cache
For NBG api portal you need 2 more headers please pay attention to the portal guides
